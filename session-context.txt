## PolicyProduct Decision

PolicyProduct is a lookup table.
Initial products seeded:
- CAR
- HEALTH
- LIFE
- HOME
- TRAVEL

No product-management API.
New products can later be inserted in DB without changing the Policy schema/code entity model.

## Validation Decisions

DTOs use DataAnnotations and IValidatableObject.

Moved validation into DTOs:
- StartDate before EndDate
- PremiumAmount positive
- CustomerId required for issue policy
- Cancellation reason not whitespace

PolicyService no longer does date/premium validation.

## Error Handling

Global handler:
src/InsuranceManagement.Api/Infrastructure/ErrorHandling/GlobalExceptionHandler.cs

It handles:
- NotFoundException -> 404
- ConflictException -> 409
- BadHttpRequestException -> 400
- JsonException -> 400
- DbUpdateConcurrencyException -> 409
- SQLite constraint violation for Customers.Email -> 409
- SQLite constraint violation for Policies.PolicyNumber -> 409
- general SQLite constraint violation -> 409
- unexpected -> 500

Program.cs configures:
- AddExceptionHandler<GlobalExceptionHandler>()
- AddProblemDetails()
- ApiBehaviorOptions.InvalidModelStateResponseFactory for consistent validation errors

Remaining service exceptions are business checks that need loaded domain state:
- missing customer/policy/product
- inactive customer/product
- cannot deactivate customer with active policies
- cannot update cancelled policy
- cannot cancel already cancelled policy

## Services

CustomerService:
src/InsuranceManagement.Api/Application/Customers/CustomerService.cs

Supports:
- Create customer
- Get all
- Get by ID
- Update
- Deactivate
- Get customer policies

Uses AppDbContext directly. No repository layer.

Important helper:
GetCustomerOrThrowAsync(Expression<Func<Customer,bool>> predicate, CancellationToken, bool asNoTracking = true)

PolicyService:
src/InsuranceManagement.Api/Application/Policies/PolicyService.cs

Supports:
- Issue policy
- Get all/filter
- Get by ID
- Update
- Cancel

Important helpers:
- BuildPolicyQuery(PolicyFilter? filter = null, bool asNoTracking = true)
- GetPolicyOrThrowAsync(IQueryable<Policy> query, CancellationToken)
- ValidateIssueData(...)
- GeneratePolicyNumber()
- MapPolicy(...)

IssueAsync was optimized:
- customer/product validation data is fetched in one read query
- still needs separate SaveChangesAsync to insert policy

CancelAsync:
- still does one tracked read and one save, because method returns updated PolicyResponse.
- Avoid raw SQL/provider-specific one-command update unless user explicitly wants that tradeoff.
